# R&D Knowledge Graph & Search Engine

## Структура проекта: что за что отвечает

### 1. Бэкенд (`api/`)
*   `dependencies.py` — Управление асинхронными сессиями базы данных SQLAlchemy.
*   `main.py` — Главный файл бэкенда (инициализация FastAPI, автоматическое создание таблиц при старте, подключение роутера).
*   `routes.py` — Определение API-эндпоинтов, включая основной метод `POST /ask` для обработки запросов пользователя.

### 2. Алгоритмическое ядро (`core/`)
*   `schemas.py` — Pydantic-модели данных (строгая валидация фактов, параметров и типов данных перед записью в БД).
*   `prompts.py` — Шаблоны системных инструкций для ИИ (для извлечения фактов, декомпозиции сложных вопросов, генерации SQL и синтеза ответов).
*   `llm_client.py` — Асинхронный HTTP-клиент для взаимодействия с API GigaChat (работает по протоколу OAuth 2.0).
*   `extractor.py` — Конвертер сырого неструктурированного текста в JSON-объекты согласно схемам Pydantic.
*   `retriever.py` — Мозг поискового движка. Выполняет декомпозицию запроса, трансляцию в SQL, а в случае отсутствия точных совпадений автоматически запускает резервный текстовый поиск по корням слов.
*   `logger.py` — Централизованное логирование событий с записью в консоль и автоматическим созданием файлов в `logs/` по таймстампу.

### 3. Слой хранения данных (`db/`)
*   `database.py` — Настройка асинхронного подключения к СУБД SQLite через драйвер `aiosqlite`.
*   `models.py` — Реляционные модели таблиц SQLAlchemy (`facts` и `parameters`) с поддержкой индексации и колонок в нижнем регистре (`_lower`) для поиска на кириллице.
*   `crud.py` — Логика записи извлеченных фактов и параметров в базу данных.

### 4. Веб-интерфейс (`frontend/`)
*   `app.py` — Главный экран приложения (чат-интерфейс ассистента).
*   `pages/dashboard.py` — Аналитическая панель (метрики по базе данных, графики популярных металлов и процессов).
*   `pages/gaps_in_knowledge.py` — Панель автоматического поиска пробелов в изученности материалов (дефицитные процессы, слабая география исследований).
*   `pages/knowledge_graph.py` — Интерактивная визуализация семантической сети графа связей между металлами, процессами и документами-источниками.

### 5. Обработка документов (`parsers/`)
*   `document_parser.py` — Чтение PDF-файлов и извлечение сырого текстового слоя (библиотека PyMuPDF).
*   `text_chunker.py` — Нарезка длинных текстов на перекрывающиеся фрагменты (чанки) для подачи в ИИ.

### 6. Вспомогательные скрипты (`scripts/`)
*   `ingest_data.py` — ETL-скрипт для пакетного обхода папки документов, чанкинга, отправки в ИИ и сохранения в БД.
*   `check_db.py` — Утилита быстрого вывода списка всех файлов, успешно импортированных в базу знаний.
*   `test_db.py` — Быстрый отладочный вывод первых 5 фактов из базы данных для проверки структуры.

## 6. Тестирование (`tests/`)
*   `conftest.py` — Общие фикстуры pytest: in-memory БД, асинхронные сессии, моки LLM и логгера.
*   `test_schemas.py` — Тесты Pydantic-схем на валидацию полей, опциональные параметры и отклонение некорректных данных.
*   `test_database_crud.py` — Тесты функций записи в БД: сохранение фактов, параметров, автозаполнение `_lower`-колонок и связывание через `fact_id`.
*   `test_text_chunker.py` — Тесты нарезки текста на чанки: проверка overlap, обработка коротких/пустых текстов и граничных случаев.
*   `test_extractor.py` — Тесты извлечения фактов из текста с моками LLM: парсинг JSON, устойчивость к сломанным ответам и невалидным схемам.
*   `test_retriever.py` — Тесты поискового движка: выполнение SQL, защита от инъекций, декомпозиция запросов, fallback-поиск и форматирование результатов.

### 8. Корневые файлы проекта
*   `run.sh` — Bash-скрипт для автоматического запуска всей системы.
*   `Dockerfile` — Инструкция по сборке Docker-образа приложения.
*   `docker-compose.yml` — Файл развертывания мультиконтейнерного окружения для запуска бэкенда и фронтенда одной командой.
*   `knowledge_base.db` — Файл базы данных SQLite, содержащий все успешно извлеченные факты.
*   `app.log` — Локальный файл логов.
*   `pytest.ini` — Конфигурационный файл pytest с настройкой асинхронного режима и путей поиска модулей.

## Как запустить проект локально

### Шаг 1. Настройка окружения
Создайте в корне проекта файл `.env` и пропишите в него ваши ключи GigaChat API (или используйте .env предоставленный в репозитории):
```env
GIGACHAT_CLIENT_ID=ваш_client_id_сбер_студии
GIGACHAT_CLIENT_SECRET=ваш_client_secret_сбер_студии
GIGACHAT_SCOPE=GIGACHAT_API_PERS
```

### Шаг 2. Запуск проекта одной командой
Запустите скрипт автоматической сборки:
```bash
chmod +x run.sh
./run.sh
```

### Ручной запуск (альтернатива)
Если нужно запустить части проекта отдельно:

1. **Запуск импорта данных (ETL):**
```bash
python -m scripts.ingest_data
```
> **ВНИМАНИЕ:** Процесс первоначального импорта данных может занять длительное время из-за большого объема анализируемых документов.

2. **Запуск API бэкенда:**
```bash
uvicorn api.main:app --host 127.0.0.1 --port 8000
```

3. **Запуск фронтенда Streamlit (в новом окне терминала):**
```bash
streamlit run frontend/app.py --server.port 8501
```

Перейдите по адресу: **`http://localhost:8501`**

## Как запустить проект через Docker

Перед запуском убедитесь, что у вас установлен Docker и запущен Docker Desktop, а также настроен файл `.env`.

### 1. Сборка и запуск контейнеров
Запустите сборку и старт приложения одной командой в терминале:
```bash
docker-compose up --build
```

### 2. Использование приложения
Docker автоматически поднимет все компоненты системы. Перейдите по адресу:
**`http://localhost:8501`**

### 3. Остановка контейнеров
Чтобы остановить работу приложения в Docker, нажмите в терминале **`Ctrl + C`** или выполните команду:
```bash
docker-compose down
```

## Как запустить тесты

Перед запуском убедитесь, что активировано виртуальное окружение `(venv)` и установлены зависимости для тестирования:
```bash
pip install pytest pytest-asyncio sqlalchemy aiosqlite pydantic httpx
```

### 1. Запуск всех тестов
Выполните команду из корня проекта, чтобы запустить весь набор из тестов с подробным выводом:
```bash
pytest tests/ -v
```

### 2. Запуск тестов отдельного модуля
Чтобы протестировать только конкретную часть системы (например, только поисковый движок), укажите путь к файлу:
```bash
pytest tests/test_retriever.py -v
```

### 3. Запуск с детальным выводом ошибок
Если какой-то тест упал, используйте флаг `--tb=short` для получения компактного трейсбека с точной строкой падения:
```bash
pytest tests/ -v --tb=short
```

### 4. Тихий режим
Для быстрого прогона без лишнего вывода (только итоговая статистика PASS/FAIL):
```bash
pytest tests/ -q
```
